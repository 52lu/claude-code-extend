# Coder Obsidian Docs

Obsidian文档编写技能，通过四步工作流生成高质量Obsidian笔记。

## 配置管理

本技能的输出路径通过配置文件管理，首次使用时引导用户设置，后续自动读取不再询问。

配置文件路径: `~/.claude/skills/coder-obsidian-docs/config.json`

配置文件格式:
```json
{
  "doc_root": "/path/to/obsidian/docs",
  "canvas_root": "/path/to/obsidian/canvas"
}
```

**注意**: `canvas_root` 和 `doc_root` 必须是不同的目录，canvas 文件独立存放。

### 配置读取逻辑

执行 Step 3 和 Step 4 前，检查配置文件：

1. **配置文件不存在或字段缺失**: 使用 `AskUserQuestion` 逐项要求用户输入路径，保存到配置文件后继续
2. **配置文件存在且完整**: 直接使用配置的路径，不再询问
3. **用户主动要求修改路径**: 更新配置文件中的对应字段，继续使用新路径

### 修改配置

当用户说"修改文档路径"、"修改canvas路径"、"更改输出目录"等时，使用 `AskUserQuestion` 收集新路径，更新配置文件。

## 工作流

```
用户请求 → Step 0 需求澄清（条件触发） → Step 1 头脑风暴 → Step 2 文档编写 → Step 3 风格转换 → Step 4 文档输出
```

### Step 0: 需求澄清（条件触发，优先于 Step 1）

收到用户请求后，**先判断是否完全理解了用户意图**，再决定是否进入 Step 1。

**「不能完全理解」的判定标准（满足任一即触发）：**

- 主题不明确：用户用"那个东西"、"最近挺火的"、"你懂的"等指代不清的表述，无法锁定具体主题
- 范围不清晰：无法判断文档要覆盖哪些内容、边界在哪里（写多深、写多宽）
- 受众/目的不明：不知道文档给谁看、用来做什么（学习参考？团队沉淀？分享科普？）
- 存在歧义：用户的需求存在两种以上合理解读，无法确定是哪一种
- 关键信息缺失：缺少动笔所需的核心要素（如主题名称、所属领域等）

**触发后的处理（必须按此顺序，不得跳过）：**

1. **调用 `grill-me` skill** 对用户展开追问式访谈，逐层澄清上述不明确的点。**不要自行假设主题、不要随手罗列一张确认表格代替访谈**——`grill-me` 的层层追问比一次性列清单更能逼出真实需求。
2. 访谈完成后，输出一份明确的需求摘要（主题、范围、受众、目的）供用户确认。
3. 用户确认后，带着澄清结果进入 Step 1 头脑风暴。

**不触发的情况：** 用户请求明确、要素齐全（主题清晰、范围可推断、受众/目的明确）时，直接进入 Step 1，不要为走流程而强行发起 grill-me 访谈。

### Step 1: 头脑风暴

调用 `brainstorming` skill，与用户确认以下内容：

- **文档主题**: 核心主题和覆盖范围
- **目标读者**: 文档面向的受众
- **文档风格**: 技术深度、语言风格（中文/英文）、正式程度
- **结构大纲**: 章节划分和重点内容
- **是否需要流程图**: 识别哪些部分适合用可视化表达

完成头脑风暴后，输出确认的文档大纲供用户审核。

### Step 2: 文档编写

调用 `documentation-writer` skill，基于Step 1确认的大纲编写文档内容。

编写要求：
- 遵循 Diátaxis 文档框架原则
- 内容准确、结构清晰、示例具体
- 使用中文编写（除非用户另有要求）

### Step 3: 风格转换

将Step 2产出的内容转换为Obsidian格式：

1. **调用 `obsidian-markdown` skill** 转换文档格式，包括：

   - **添加 frontmatter 笔记属性**：每篇笔记的 frontmatter 必须包含以下五个字段，缺一不可，不得增删、改名或使用别名（如不得用 `aliases` 代替 `classify`）：

     | 字段 | 含义 | 是否可多值 | 取值要求 |
     |------|------|-----------|----------|
     | `tags` | 标签 | ✅ 可多个 | 主题相关标签，使用 `tag/subtag` 层级形式 |
     | `classes` | 分类 | ❌ 单值 | 文档所属的一级分类，如 `技术笔记`、`学习笔记`、`项目文档` |
     | `classify` | 别名 | ✅ 可多个 | 笔记的别名，用于 Obsidian 中以其他名称检索到本笔记 |
     | `date` | 日期 | ❌ 单值 | `YYYY-MM-DD` 格式，取文档编写当天日期 |
     | `path` | 路径 | ❌ 单值 | 笔记在 Obsidian 库中的归属路径（相对路径），如 `4.AIOutput/Git` |

     **frontmatter 标准模板（直接套用，多值字段用 YAML 列表语法）：**
     ```yaml
     ---
     tags:
       - <标签1>
       - <标签2>
     classes: <分类>
     classify:
       - <别名1>
       - <别名2>
     date: YYYY-MM-DD
     path: <相对路径>
     ---
     ```

     **字段填写说明：**
     - `tags` 与 `classify` 为可多值字段，**必须使用 YAML 列表（`- 值`）语法**，即使只有一个值也用列表形式，便于后续追加
     - `classes`、`date`、`path` 为单值字段，直接写值，不要用列表
     - 五个字段顺序建议保持模板顺序（tags → classes → classify → date → path）

   - **在 frontmatter 之后、正文内容之前，插入以下警示 callout**（所有文档必须包含）：
     ```markdown
     > [!warning] ⚠️  温馨提示
     > <font color=red>**我已经帮您完成了文档编写，但我无法替你吸收和学习,请保持持续进化,勿做工具人！**</font>
     ```
   - 使用 `[[wikilinks]]` 替代普通链接
   - 使用 `> [!note]` 等 callout 语法
   - 使用 Obsidian 特有的嵌入语法

2. **流程图处理**：根据流程复杂度选择方案：

   | 场景 | 方案 | 说明 |
   |------|------|------|
   | 简单流程（节点少、线性走向） | Mermaid | 直接内嵌在文档中，轻量直观 |
   | 复杂流程（多分支、多层级、网状关系） | json-canvas | 调用 `json-canvas` skill，独立 canvas 文件 |
   | 不确定复杂度 | 询问用户 | 向用户说明两种方案优劣，由用户决定 |

   - 使用 Mermaid 时：直接在文档中写入 ` ```mermaid ``` ` 代码块
   - 使用 json-canvas 时：
     - 读取配置中的 `canvas_root` 作为保存目录（未配置则询问用户，保存后不再询问）
     - canvas 文件保存到 `canvas_root`
     - 在文档中使用 `![[filename.canvas]]` 嵌入引用

### Step 4: 文档输出

1. 读取配置中的 `doc_root` 作为文档根目录（未配置则询问用户，保存后不再询问）

2. 在 `doc_root` 下查找与文档主题匹配的子目录

3. 向用户确认目标子目录和文件名

4. 保存文档到确认的路径

5. 文档名称格式: '编号.名称.md ⚠️  '; 编号的生成规则参考所在目录的其他文件，不能重复，应该是递增; 如果已经存在的文件不符合格式，则需要修改

## 执行规范

- 每个Step完成后向用户汇报当前进度
- Step 0 的需求澄清结果必须经用户确认后才进入 Step 1
- Step 1 的头脑风暴结果必须经用户确认后才进入Step 2
- Step 4 的目标目录必须经用户确认后才保存文件
- **Step 0 强制规则**：判定为「不能完全理解」时，必须先调用 `grill-me` skill 访谈澄清，**不得**用以下方式替代：自行假设主题后直接动笔、随手罗列一张确认表格代替访谈、把模糊需求原样带进 Step 1 头脑风暴。
- **frontmatter 自检（Step 3 完成后强制执行）**：保存文档前核对 frontmatter 是否同时包含 `tags`、`classes`、`classify`、`date`、`path` 五个字段；`tags` 与 `classify` 是否为 YAML 列表形式（即使单值也用列表）；是否误用了 `aliases` 等非标准字段名。任一项不满足则修正后再保存。
